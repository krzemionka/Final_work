How to write API documentation?

When you are about to write API documentation it is advisable to follow a plan:

source: own

Let’s discuss them as following above listed steps as they will help you to organize your work, not only in API documentation but actually in most technical writing tasks.

Planning is about creation of a documentation plan, even if it’s brief or incomplete. Find answers to questions like:

  • When is the release for the new API product?
  • What content resources are already available? Are there any drafts or product design documents?
  • Where will the docs be published and what outputs and formats will be needed?
  • Whom can I contact for information, review and sign off on the content? Identify the product manager, tech lead, engineers, QA, and other stakeholders.
  • Who is the recipient? What do you want readers to be able to do?

If you don’t know where to start ask the engineers to write a first draft or give them a template with some fill-in-the-blank type sections. It might be your starting point so that you could spend time on the content and trying to understand whether it’s coherent for the audience.

Now it is time to gather all necessary material. Get API description, the used language, other references and sample cases from the API developers. Make a detailed research of the already existing documents. Even if the product is totally new, some engineer had to write a description of the product feature to describe the problem and the need for the API. Also, there should be another document called “a product design document” that acts as a blueprint for the feature’s design. You may also have a look at reference documentation where engineers make annotations in the source code. Gather as many documents as you can find, list them in your doc plan and then go through these documents to learn about the product and see what material you can use in your docs. It would be great to see a demo of the product. It might be just a scenario of an API call and its response. Most engineers are proud of what they had built so they will be happy to present the demo.

 

Our next step deals with information gathering.

source: own

Think of yourself as a technical journalist who gathers information for their story by interviewing people. You need to adjust your questions to several different groups of people:

  • Engineering — the group that coded the feature
  • Product management — the group that oversees the feature’s planning and strategy
  • Partner engineers — the group that helps users implement the feature
  • Quality assurance — the group that tests the feature

Reaching so many different groups gives you the chance to see the product from different perspectives. Remember to be ready for each meetings - prepare lists of questions and send them before the meeting. It will show that you value your partner’s time and are serious about your job. When you meet, it is advisable that you record your interview, especially with engineers. Some exemplary questions to ask the engineers:

  • Are there any requirements to use this API?
  • How do you set it up?
  • How do you initialize it?
  • Can you talk through the code in the sample app?
  • How do you use this API for some common use cases?
  • Are there limitations that partners will want to know about?

Interviewing a product manager (PM) may also be beneficial because this person is focused on different aspects of the product than technical ones (like product messaging, use cases, audience analysis and concepts related to the product). But also PM will be fond of presenting the product in an positive light and as a great success. Some exemplary questions to ask the PM:

  • Who is the audience for this API?
  • What are some use cases for this API?
  • What are some important concepts to know when working with this API?
  • When is the release date for the feature?

When dealing with the group of partner engineers/solutions engineers have in mind that they are the most important group as they interface with partners/users to help them implement the company’s products and they are able to provide you with the closest feedback from users. What is more, partner engineers are fond of a product documentation. The better documentation the less work they have with product implementation and answering users’ questions, troubleshoot issues, etc.

  • Which partners are we developing this feature for/with?
  • How is this target partner planning to use the API?
  • How does this partner implement the feature within their company? What are their workflow and setup?
  • What issues or questions do you anticipate partners will have related to this API?

Another group to interview is the QA group. They make a list of test cases that they are testing the feature against so you will information about how to test the API.

  • Do you have a list of test cases that you’re testing the API against?
  • How are you testing the API? Is there a specific test environment you’re using?
  • Are there any test devices or setup I can use?
  • Is there a sample app you’re using to do the testing? How do you use this sample app?
  • Do you have any documents that describe testing steps?
  • Are there any bugs or limitations to be aware of? Where are the bugs tracked?
  • Do you have a list of what data the API should return in various scenarios?
  • Are there any resources related to this API that might be helpful?

 

source: own

Most probably now you have a long document with many pages. Everything looks disorganized and reminds of a pile of information rather than a nicely prepared document. Take all the information and group it into different sections so that you end up with 4-5 sections per document. An example of sections may look like this:

  • Product overview — high-level description of the feature, requirements, prerequisites, geographic availability, components involved, workflow implementation, how to get help, etc.
  • Concepts — foundational concepts related to using the feature. These topics might include diagrams or other visuals to assist with understanding.
  • Reference — reference documentation.
  • How-tos — descriptions of how to implement the API and handle various scenarios.

Fill each section with appropriate content. Now you need subject matter experts to review what you've written. Then you need to make the content more visual. Add screenshots, video tutorials, concept diagrams and other illustrations to your content to make it understandable and clear. As soon as the the content is more or less finalized, run it through a spell checker: Grammarly, Microsoft Word, Google Docs, HyperSTE plugin (to check for simplified technical English) or Acrolinx.

source: own

Document review is a must because you’re usually writing about a technical domain that you’re not an expert in. One way to gather people’s feedback is to organize a review meeting. However, people are usually not fond of reading and making corrections so you might organize a review meeting and choose one of the following techniques:

  • split the meeting into the reading part where everyone reads your document and the second part, when you collect all comments
  • ask people to comment on while reading the document

It would be advisable not to review more than the reviewer can read in 20 minutes which is about 6 pages. When there is a lot of content to review try to set up multiple doc review sessions. If the reviewers run out of comments ask questions and dispel any doubts you have. After the meeting, incorporate edits into the documentation. If you don’t incorporate someone’s feedback, it would be a good practice to indicate a reason in your revision notes.

Another strategy is to layer the review process so each reviewer analyse your content one by one. You have an opportunity to incorporate the edits from each review before the next reviewer sees the content. You can improve the content so as to make your document better.

When it comes to organising a documentation review meeting, avoid Mondays mornings or Friday afternoons. Also lunchtime is not a good time to gather people

The review cycle ends when you release the product.
